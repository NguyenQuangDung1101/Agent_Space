## 1. Core entities

```text
Conversation
    ├── Contains multiple user/assistant messages
    ├── Contains multiple execution Sessions
    ├── Stores shared conversation memory
    └── Must be created before sending tasks

Session
    ├── Represents one user task execution
    ├── Belongs to exactly one Conversation
    ├── Contains analysis, plan, executions, events, and final response
    └── May pause and resume when waiting for user input

Message
    ├── Belongs to a Conversation
    ├── May trigger a new Session
    ├── May answer a pending UserContactRequest
    └── May contain document attachments

ConversationMemory
    ├── Stores a compact summary of previous sessions
    ├── Includes important user requests, decisions, plans, and responses
    └── Is provided as context to new sessions
```

A normal user message creates a new Session.

A message answering a pending user-contact request resumes the existing Session instead of creating another Session.

## 3. Conversation and session flow

```text
Create Conversation
    ↓
User sends message
    ↓
Manager saves message to Conversation
    ↓
If message answers a pending UserContactRequest:
    Resume the existing Session
Else:
    Create a new Session
    ↓
Load ConversationMemory
    ↓
Analyze Agent
    ├── Single execution
    └── Team execution
    ↓
Execute selected solution
    ↓
Save final response as an assistant message
    ↓
Update ConversationMemory
```

The detailed analysis, team formation, tool results, and execution events stay inside the Session.

Conversation memory stores only the useful shared summary needed by later tasks.

## 4. User-contact flow

This is a method for every agent to interact with user in mid-task
User contact must be separate from direct or broadcast team communication because it pauses execution.

An agent may emit:

```text
<user_contact>
{
  "question": "Please review the proposed import mapping.",
  "reason": "Confirmation is required before modifying the Knowledge graph.",
  "expected_response": "Free-text review or approval"
}
</user_contact>
```

Flow:

```text
Agent raises UserContactRequest
    ↓
Orchestrator sends it to Manager
    ↓
Session status becomes WAITING_FOR_USER
    ↓
Save active agent state and checkpoint
    ↓
Emit user_contact_requested event
    ↓
GUI displays the agent's question
    ↓
User sends a free-text response
    ↓
Response is linked to the contact request
    ↓
Session status returns to RUNNING
    ↓
Resume the same agent instance and task
```

The user may respond naturally, for example:

```text
The plan is acceptable, but merge duplicate company nodes before importing.
```

The agent may continue, revise its plan, or raise another user-contact request.

For team execution, the whole Session pauses while waiting for the user.

## 5. Graph data model

Two graph domains exist in the same Neo4j database.

### Agent Space domain

```text
(System)-[:REGISTERS]->(Agent)
(System)-[:REGISTERS]->(Tool)

(Conversation)-[:HAS_MESSAGE]->(Message)
(Conversation)-[:HAS_SESSION]->(Session)
(Conversation)-[:HAS_MEMORY]->(MemorySnapshot)

(Session)-[:TRIGGERED_BY]->(Message)
(Session)-[:HAS_TASK]->(Task)
(Session)-[:HAS_ANALYSIS]->(Analysis)
(Task)-[:SOLVED_BY]->(Solution)

(Solution)-[:USES_AGENT]->(Agent)
(Solution)-[:USES_TOOL]->(Tool)

(Session)-[:HAS_EXECUTION]->(Execution)
(Execution)-[:HAS_AGENT_RUN]->(AgentRun)
(AgentRun)-[:INSTANCE_OF]->(Agent)
(AgentRun)-[:HAS_TOOL_RUN]->(ToolRun)

(Session)-[:HAS_CONTACT]->(UserContact)
(UserContact)-[:REQUESTED_BY]->(AgentRun)
(UserContact)-[:ANSWERED_BY]->(Message)
```

### External Knowledge domain

```text
(Knowledge {id: "external_knowledge"})
    -[:HAS_DOCUMENT]->(KnowledgeDocument)

(KnowledgeDocument)-[:HAS_CHUNK]->(KnowledgeChunk)
(Knowledge)-[:HAS_ENTITY]->(KnowledgeEntity)

(KnowledgeChunk)-[:MENTIONS]->(KnowledgeEntity)
(KnowledgeEntity)-[:RELATED_TO]->(KnowledgeEntity)
(KnowledgeEntity)-[:SUPPORTED_BY]->(KnowledgeChunk)
```

The main Agent Space repository must ignore the external Knowledge domain during normal system discovery and execution queries.

Only explicit Knowledge agents access it.

Optional provenance links may connect the domains:

```text
(AgentRun)-[:IMPORTED]->(KnowledgeDocument)
(AgentRun)-[:RETRIEVED_FROM]->(KnowledgeEntity)
```

These links record provenance but do not make external knowledge part of the normal Agent Space control graph.

```text
Session statuses:
    PENDING
    RUNNING
    WAITING_FOR_USER
    COMPLETED
    FAILED

Additional events:
    conversation_created
    message_received
    session_started
    analysis_started
    analysis_completed
    planning_started
    planning_completed
    agent_running
    tool_running
    user_contact_requested
    user_contact_answered
    session_resumed
    synthesis_started
    session_completed
    session_failed
```

A Session is completed only after its final response is saved and added to the Conversation.
